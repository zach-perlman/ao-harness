# Activation Oracles

This repository contains the code for the [Activation Oracles](https://arxiv.org/abs/2512.15674) paper.

## Overview

Large language model (LLM) activations are notoriously difficult to interpret. Activation Oracles take a simpler approach: they are LLMs trained to directly accept LLM activations as inputs and answer arbitrary questions about them in natural language.

## Installation

```bash
uv sync
source .venv/bin/activate
huggingface-cli login --token <your_token>
```

## Pre-trained Models

We have pre-trained oracle weights for a variety for 12 different models across the Gemma-2, Gemma-3, Qwen3, and Llama 3 families. They are available on Hugging Face: [Activation Oracles Collection](https://huggingface.co/collections/adamkarvonen/activation-oracles)

The wandb eval / loss logs for these models are available [here](https://api.wandb.ai/links/adam-karvonen/cu11tv7r). Note that the smaller models (1-4B) tend to have worse OOD eval performance, so I'm not sure how well they will work.

## Training

Training is config-driven. You define a training config JSON, then run dataset generation and training as separate steps.

### 1. Define a training config

Training configs live in `training_configs/` and are generated by `utility_scripts/draft_training_configs.py`. A config specifies the model, dataset mixture, hyperparameters, and (optionally) a HuggingFace repo to push the trained LoRA to.

Pre-made configs:
- `training_configs/quick.json` — Small quick run for debugging (Qwen3-8B, ~96 steps)
- `training_configs/e2e_test.json` — Tiny config for the regression test (Qwen3-0.6B)

To generate configs for all models in the HuggingFace collection:

```bash
python -m utility_scripts.draft_training_configs
```

### 2. Generate datasets

Dataset generation uses vLLM and must run as a single process (before `torchrun`):

```bash
python nl_probes/sft.py --config training_configs/quick.json --gen-only
```

### 3. Train

Training uses `torchrun` for DDP:

```bash
torchrun --nproc_per_node=<NUM_GPUS> nl_probes/sft.py --config training_configs/quick.json
```

To push the trained LoRA to HuggingFace, pass `--hf-repo-id`:

```bash
torchrun --nproc_per_node=1 nl_probes/sft.py --config training_configs/quick.json --hf-repo-id adamkarvonen/my-lora
```

### 4. Evaluate

Run open-ended evals on a trained LoRA:

```bash
python -m experiments.run_all_open_ended_evals --verbalizer-lora adamkarvonen/my-lora
```

Without `--verbalizer-lora`, this evaluates the standard pre-trained LoRAs.

### End-to-end script

For convenience, `scripts/e2e_train_eval.py` runs all steps (gen, train, push, eval) in sequence:

```bash
# Quick debugging run
python scripts/e2e_train_eval.py --quick --hf-repo-id adamkarvonen/ao-e2e-test

# Full training with a config
python scripts/e2e_train_eval.py --config training_configs/my_config.json --hf-repo-id adamkarvonen/my-lora
```

Options: `--skip-gen`, `--skip-eval`, `--num-gpus N`.

### Model-specific notes

- **Gemma models**: Set `export TORCHDYNAMO_DISABLE=1` before training.
- **Large models (32B+)**: Set `"load_in_8bit": true` in the training config to use 8-bit quantization.

### Training tasks

A full training run uses a diverse mixture of:
- System prompt question-answering (LatentQA)
- Binary classification tasks
- Self-supervised context prediction (PastLens)

You can train any model available on HuggingFace Transformers by setting the model name in the config.

## Running on the Slurm Cluster (RunPod)

For full infrastructure details, see `RUNPOD_INFRA_GUIDE.md` and `RUNPOD_CLUSTER_INFO.md`.

### Initial Setup

```bash
# SSH to your login node (e.g. node-25 for usernames starting with A-B)
ssh your_alias

# Work from /workspace-vast/<username>/, not /home/<username>/
cd /workspace-vast/$(whoami)/activation_oracles_dev

# Run setup (installs deps, logs into wandb/HF, downloads model)
./setup.sh
```

### Submitting Jobs

Job scripts live in `jobs/`. Submit from the repo directory:

```bash
cd /workspace-vast/$(whoami)/activation_oracles_dev
sbatch jobs/quick_test.sh
```

Example job script (`jobs/quick_test.sh`):
```bash
#!/bin/bash
#SBATCH --job-name=ao_quick_test
#SBATCH --partition=general
#SBATCH --qos=high              # high = can't be preempted; low = can be preempted
#SBATCH --gres=gpu:1
#SBATCH --cpus-per-task=8
#SBATCH --mem=32G
#SBATCH --time=02:00:00
#SBATCH --output=logs/%x_%j.out
#SBATCH --error=logs/%x_%j.err

set -euo pipefail
cd /workspace-vast/$(whoami)/activation_oracles_dev
source .venv/bin/activate
source .env

CONFIG=training_configs/quick.json

python nl_probes/sft.py --config "$CONFIG" --gen-only
torchrun --nproc_per_node=1 nl_probes/sft.py --config "$CONFIG"
```

### Monitoring Jobs

```bash
# Check job status
squeue -u $USER

# Watch output live (stdout + stderr)
tail -f logs/<jobname>_<jobid>.out logs/<jobname>_<jobid>.err

# Job details
scontrol show job <jobid>

# Cancel a job
scancel <jobid>
```

### GPU and Cluster Status

```bash
# Check GPU availability across nodes
sinfo -o '%n %G %C %T'

# Check GPU usage on a specific node
ssh <node> nvidia-smi

# See who's using GPUs
squeue -o "%u %T %b" | grep RUNNING | sort | uniq -c
```

### QoS (Priority)

- `--qos=high` — Cannot be preempted; can preempt low-priority jobs
- `--qos=low` — Default; can be preempted (killed) by high-priority jobs

### Interactive Sessions

```bash
# Basic interactive session on dev partition (1 GPU)
srun -p dev,overflow --qos=dev --job-name=D_$(whoami) --gres=gpu:1 --cpus-per-task=8 --mem=32G --pty bash

# Prefix job name with D_ for automatic midnight cleanup
```

### Storage

- `/workspace-vast/<user>/` — Persistent shared storage (survives restarts, visible from all nodes)
- `/workspace/<user>/` — Fast temporary storage (use for training checkpoints, clean up after)
- `/home/<user>/` — Small home directory

### Important Rules

⚠️ DO NOT EVER EXPORT CUDA_VISIBLE_DEVICES YOURSELF. SLURM DOES THIS FOR YOU ⚠️
    If you do it will cause slurm to land jobs on GPUs that might be utilised causing all jobs to drain into that slot and crash!
⚠️ DO NOT SET `--mem=0` IN YOUR SLURM JOB CONFIG ⚠️. This will request ALL CPU memory. Slurm manages both CPU and GPU resources. Currently, we do not limit the amount of CPU memory you can request, so accidentally requesting all available CPU memory can prevent other jobs from being allocated. We suggest only requesting at most 128G CPU memory per GPU you request, unless you have a good reason to do otherwise.
- **Use `sbatch`/`srun`** to run GPU work, not direct execution on login nodes
- **Clean up** training checkpoints from `/workspace/` after jobs complete

## Generating Synthetic Training Data on the Cluster

The synthetic data pipeline lives in `data_pipelines/training_data/`. It streams WildChat prompts, generates model responses and continuations via vLLM, then (optionally) uses Sonnet to create QA pairs.

For the full runbook, see [docs/generating_training_data.md](docs/generating_training_data.md).

### Sharded generation (stages 1-2, vLLM)

```bash
cd /workspace-vast/$(whoami)/activation_oracles_dev
source .env

RUN=synthetic_qa_400k

# 1. Prep prompts (CPU only, run on login node)
.venv/bin/python data_pipelines/training_data/generate_training_data.py prep --run "$RUN" --n-prompts 400000

# 2. Submit sharded vLLM jobs (4 GPUs, 1 per shard)
sbatch data_pipelines/training_data/submit_training_data.sh

# 3. After all jobs finish, merge shards
.venv/bin/python data_pipelines/training_data/generate_training_data.py merge-stage12 --run "$RUN" --num-shards 4
```

To adjust the run name or number of shards/GPUs, edit `submit_training_data.sh` (`RUN`, `--array`, and `NUM_SHARDS`).

### QA generation (stage 3, Sonnet API)

After merging, run stage 3 to generate QA pairs. For large runs (>10K entries), use the **Batch API** — it has much higher throughput and doesn't require managing concurrency:

```bash
# Batch API (recommended for large runs)
source .env
.venv/bin/python data_pipelines/training_data/generate_training_data.py stage3 --run "$RUN" --batch
```

**Important**: The `--batch` flag uses `ANTHROPIC_API_KEY_BATCH_API` from `.env`, which is a separate org/key from the regular API key. Make sure this is set.

The batch path submits all requests at once (chunked into 100K batches), polls until complete, and writes results to JSONL. Batch IDs are saved under `artifacts/<run>/stage3/batch_state.json` for inspection. Reruns skip entries already present in the JSONL, but the current code does not fully reconnect to previously submitted Anthropic batches.

For small runs or debugging, the async path is faster (results come back immediately):

```bash
.venv/bin/python data_pipelines/training_data/generate_training_data.py stage3 --run "$RUN"
```

Both paths support resume if interrupted.

### Single-GPU end-to-end (for prototyping)

```bash
.venv/bin/python data_pipelines/training_data/generate_training_data.py all --run local_debug --n-prompts 1000
```

Runs all stages on a single GPU without sharding.
